How to write code

Here are two articles I recently read, about how we should write code to prohibit bugs and make code easy to understand, that really resonated with me:

Be explicit about the order of executing: Many programs are a ordered list of statements, forming a step-by-step recipe to get work done. However, it is easy to introduce bugs and errors by unintentionally changing the order of statements. Instead, one should make dependencies explicit between the steps. When you consider the steps as nodes and the dependencies as edges, you describe a directed graph of steps. Then it should be obvious and clear which order to execute the steps are intended. Namely any order that visits and executes the steps from left to right (called topological order of the graph). What's Functional Programming All About? (Haoyi)

Refactor code like a zip compressor algorithm: During refactoring keep semantic compression in mind -- namely writing the shortest amount of code to produce the same result. Only introduce abstraction (for example functions) when they make your programs shorter and easier; remove abstractions that make the program longer and harder to understand. Semanctic Compression. Working on The Witness, Part 11 (Muratori)

Oh and by the way: The first article tells us that, independent of the language you write code in, if your code depends on the order of statements, we call it imperative and if the dependencies are explicit, we call it functional programming.

And the second article about semantic compression argues, that object-oriented code is often a competing goal with semantic compression, because thinking about objects can sometimes introduce boilerplate and abstractions that are only necessary for the objects, but not for the result.
